This page is about modding Zones (District specializations) and zone slots. These are new modding features from the Phoenix 4.0 update and do not apply to earlier versions.

Please note the contents of this page can become outdated as new patches and fixes are applied to the 4.x series of Stellaris updates.

This document may be incomplete, if you are a modder and have something to add, feel free to contribute.

From an in-game perspective, a planet's main district can have two specializations (zones) and secondary districts can have one.

From a modding perspective, Districts have zone slots in which one zone per slot can be built.

Zone slots

The code for districts references zone slots. Zone slots specify what zones (specializations) can be built on a district.

For example in the base "City" district (district_city) the slot_city_government slot permits the zone_default zone, which provides 6 buildings and allows some variety of buildings to be built in it. Highlights:

Highlights:

Every zone slot is required to specify at least one zone (specialization)

  • Zone slot localizations are not shown in the planet UI, so you can skip writing descriptions for them
  • Zone slots can specify zone sets, using included_zone_sets which is a shortcut to permit any zone to be built in that slot as long as it corresponds to the associated sets.

In the below example, slot_nexus, the code permits any zone to be built that is part of the nexus set.

slot_nexus = {
	included_zone_sets = {
		nexus
	}
	unlock = { # planet scope
		uses_district_set = machine_world
	}
}

Zones (District Specializations)

Zones (shown in-game and referred to generally as District Specializations)

The idea with zones is that they make districts quite flexible in what they produce. Previously, districts conceptually mainly produced one resource, with some secondary resources occasionally. In Phoenix 4.0 the idea is that the production of a district is mainly decided by the zone, which adds jobs of a certain category/theme, further augmented by buildings in that zone. You can change the production (jobs provided) of a district to be whatever you like, by modding the zone, or creating several different zones.

Note that districts can still produce their own resources, or add jobs as they were in 3.x.

Highlights

  • Zones have their own resources block
  • they provide building slots (up to 3 in the base game)
  • they can apply modifiers that are multiplied by the number of districts it's built on
  • can apply planet modifiers and country modifiers

District planet modifier

Zones have a modifier, district_planet_modifier which will apply the modifier, multiplied for every district that is built.

Unlike planet_modifier, which is applied to the planet only once, district_planet_modifier is applied multiple times.

Using an example from zone_urban, for every Urban district built, a static amount of housing will be added.

triggered_district_planet_modifier = {
	planet_housing_add = @zone_urban_housing_positive
}

Using zones to swap districts

Zones can be used to change the district type using the swap_type key.

Take the below zone code, for example. The swap_type key points to district_nexus_factory. If this zone is built, the district will get changed to that type.

Also, as an example of zone_set usage, the zone_factory_nexus zone can be built in any zone slot that has the nexus set (see above zone_slot example)

zone_factory_nexus = {
	inline_script = {
		script = zones/shared_industrial_factory_zone
		AMOUNT = @doubled_scaling_district_1_job
	}
	swap_type = district_nexus_factory
	swap_type_weight = 5

	zone_sets = { nexus }

	inline_script = {
		script = zones/shared_arcology_zone_modifiers
	}
}

In the target district, district_nexus_factory, the potential block should be potential = { always = no}.

This will prevent the target district from appearing on the planet in the UI in unexpected situations.

Triggered zone names

A district may have some flavor text in the planet UI just underneath the district's name.

This is accomplished by setting up a triggered_name = { } block in the district.

Triggered names resolve in order, so whichever trigger qualifies first will be the one that appears.

Example from the urban district: 

triggered_name = {
	trigger = {
		num_zones = {
			type = zone_research_unity
			value > 0
		}
		num_zones = {
			type = zone_industrial
			value > 0
		}
		exists = owner
		owner = {
			num_owned_planets = 1
		}
	}
	text = city_space_age_industry_1
}

triggered_name = {
	text = district_city
}

In the above, if there is a specific zone built, and there is also an industrial zone built, then change the flavor text to show the space age industry text.

If the first triggered_name block does not resolve, then by default the district_city localisation text will be shown.
